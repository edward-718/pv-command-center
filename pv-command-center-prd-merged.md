# PV智枢 产品需求文档（合并版 v0.2）

> 面向药物警戒从业者的学习工作类项目管理工具。本文整合原始 PRD 与方案 A 补全设计，包含已确认的待确认项、完整数据模型、API 接口定义和任务状态机，用于指导 MVP 开发与验收。

---

## 需求判定

本产品属于 0 到 1 的企业 SaaS 工具，并包含 AI 辅助能力和合规审计要求。PRD 采用中重度深度：先保证 MVP 能被开发和验收，未确认的商业、集成和算法指标已在本文中给出明确决策。

| 维度 | 判定 | 说明 |
|---|---|---|
| 需求类型 | 0 到 1 产品 / 企业 SaaS / AI 辅助工具 | 从创意方案进入产品定义，已明确用户、页面、流程、权限、验收标准。 |
| 复杂度 | 中重度 | 涉及多角色协作、监管时限、审计证据、AI 文案生成和外部系统集成。 |
| 行业 | 医疗健康 / 药物警戒 | 工作场景强依赖 SOP、质量复核、合规记录和任务追踪。 |
| 风险等级 | 高 | 不得让 AI 直接替代医学判断；权限、审计、数据安全和时限计算需要明确规则。 |
| 证据强度 | PM 假设 + 用户提供方向 | 尚未补充真实用户访谈和竞品调研，但数据模型、API 和指标体系已定义完整。 |

**已确认决策**：纯 Web MVP（移动端延后）；不接入现有安全数据库（独立系统）；首批服务对象为药企内部 PV 团队；邮件提醒延后，先做站内通知；审计包导出先 HTML 后 PDF；AI 数据源基于项目内已有内容。

## 背景与目标

PV 从业者的日常工作包含病例处理、医学审评、质量复核、监管提交、信号评估、供应商协作和审计准备。实际工作中，任务经常分散在邮件、Excel、共享盘、会议纪要和多个业务系统中，项目经理需要反复追踪责任人、时限、版本和证据。

PV智枢要解决的核心问题是：把药物警戒项目中分散的任务、文档、时限、责任人和审计记录放到同一套工作台中，让团队清楚知道"现在最该处理什么、谁负责、风险在哪里、证据是否完整"。

### 目标

- **目标 1：降低工作追踪成本** — 减少人工维护表格、邮件确认和临时找材料，把日常任务和项目状态集中管理。
- **目标 2：降低时限遗漏风险** — 围绕 Day 0、随访、质量复核和提交窗口建立提醒规则，优先暴露高风险任务。
- **目标 3：提高审计准备度** — 在工作过程中自动沉淀操作记录、审批意见、版本记录、提交回执和偏差处理记录。

## 用户与场景

| 角色 | 使用频率 | 主要痛点 | 核心诉求 | MVP 权限 |
|---|---|---|---|---|
| PV 项目经理 | 每天 | 任务多、时限紧、跨部门确认成本高。 | 快速查看风险任务、分配责任人、跟踪交付和输出周报。 | 创建项目、配置模板、分配任务、查看全部项目数据。 |
| Case Processor | 每天 | 病例材料不完整，随访和复核状态容易混乱。 | 清楚知道待处理病例、缺失信息、下一步动作和截止时间。 | 处理被分配任务、上传附件、提交复核。 |
| Safety Physician | 每周多次 | 医学判断散落在邮件和文档批注中，后续难追溯。 | 在病例或信号主题下直接给出医学意见，并保留依据。 | 查看病例详情、填写医学意见、标记需升级事项。 |
| QA / 质量团队 | 按项目阶段 | 审计前才集中追证据，偏差关闭状态不清晰。 | 持续看到证据完整度、偏差、CAPA 和 SOP 偏离风险。 | 查看审计记录、发起质量复核、导出审计包。 |
| CRO / 供应商 | 每天或每周 | 任务口径和 SLA 变化靠邮件传递，责任边界不清。 | 只看到与自己相关的任务、缺失项、交付要求和反馈意见。 | 仅访问授权项目与任务，提交交付物和说明。 |
| 系统管理员 | 按需 | 角色、项目、SOP 和模板需要统一维护。 | 配置租户、角色、模板、字典和提醒规则。 | 管理组织、角色、模板和系统配置。 |

## MVP 范围

MVP 目标是先跑通"项目创建 → 任务分配 → 时限追踪 → 协作处理 → 证据归档 → 审计导出"的主链路。聚合报告、深度信号管理和复杂系统集成可放到后续版本。

| 范围 | 包含 | 暂不包含 |
|---|---|---|
| P0 必须做 | 登录与角色权限、项目驾驶舱、PV 项目模板、任务管理、时限提醒、病例任务详情、附件与评论、审计日志、审计包导出。 | 完整 E2B 自动生成、复杂医学编码推荐、跨国法规规则库自动维护。 |
| P1 应该做 | 供应商协作空间、CAPA 任务、AI 周报草稿、会议纪要草稿、项目风险摘要。 | AI 自动医学判断、自动对外监管提交。 |
| P2 可后置 | 信号主题池、聚合报告项目管理、移动端提醒、数据看板、第三方系统 API。 | 多语言本地化、商业计费、客户自定义工作流市场。 |

## 核心流程

以下流程是 MVP 的主流程。任何新功能若不能服务这条链路，应延后到后续版本。

```
创建项目 → 生成任务 → 协作处理 → 风险提醒 → 质量复核 → 审计导出
```

1. **创建项目**：选择模板、产品、国家地区、项目类型。
2. **生成任务**：系统按模板生成节点、负责人、截止时间。
3. **协作处理**：成员处理任务、上传证据、评论和提交复核。
4. **风险提醒**：系统根据时限、状态和缺失项提示风险。
5. **质量复核**：QA 或项目经理确认结果、记录偏差。
6. **审计导出**：导出项目履历、附件清单、操作记录。

## 任务状态机

任务状态流转是 PV 项目管理的核心规则。以下状态机明确了各状态间的合法转换、触发条件和权限要求。

### 状态列表

- 未开始
- 处理中
- 待复核
- 需补充
- 已完成

### 状态转换规则

| 转换 | 触发条件 | 执行角色 | 必填字段 | 审计记录 |
|---|---|---|---|---|
| 未开始 → 处理中 | 任务负责人领取或开始处理 | 任务负责人 | 无 | 记录领取人和时间 |
| 处理中 → 待复核 | 处理完成，提交给复核人 | 任务负责人 | 提交说明（文本，必填） | 记录提交人、时间和说明 |
| 待复核 → 已完成 | 复核人确认结果通过 | PM / Physician / QA | 复核意见（文本，必填） | 记录复核人、时间和意见 |
| 待复核 → 需补充 | 复核不通过，退回处理人 | PM / Physician / QA | 退回原因（文本，必填） | 记录退回人、时间和原因 |
| 需补充 → 处理中 | 处理人补充完成后重新提交 | 原任务负责人 | 补充说明（文本，必填） | 记录补充内容和时间 |
| 处理中 → 已完成 | 特定角色直接完成任务（紧急通道） | PM / QA | 完成说明（文本，必填） | 记录操作人、时间和原因 |

**规则说明**：任务进入"待复核"后，处理人不可直接改为"已完成"；必须由有复核权限的角色确认。复核不通过则进入"需补充"，并要求填写原因。任何状态转换失败（如权限不足、必填字段缺失）必须给出明确错误提示。

## 功能需求总览

| # | 模块 | 优先级 | 功能说明 | 验收要点 |
|---|---|---|---|---|
| F1 | 登录与权限 | P0 | 支持账号登录、角色分配和租户内数据隔离。不同角色只能看到授权项目、任务、附件和审计记录。 | 普通成员不能查看未授权项目；供应商只能访问被邀请的项目空间。 |
| F2 | 项目驾驶舱 | P0 | 展示项目总览、待办任务、临近时限、高风险事项、证据完整度和供应商交付状态。 | 进入首页后 5 秒内能定位今日最紧急任务和责任人。 |
| F3 | 项目模板 | P0 | 支持创建 ICSR、监管问询、CAPA 等项目模板。模板包含任务节点、默认角色、相对截止时间和必填证据。 | 选择模板后能自动生成项目任务；模板字段可编辑但有必填校验。 |
| F4 | 任务管理 | P0 | 支持任务创建、分配、状态流转、截止时间、优先级、关联病例编号、附件、评论和复核。 | 状态流转符合任务状态机；逾期任务在列表中突出显示。 |
| F5 | 时限提醒 | P0 | 根据任务截止时间和风险等级生成提醒。MVP 先支持站内通知，邮件提醒延后。 | 临近截止、逾期、复核退回均能生成通知；重复提醒可配置。 |
| F6 | 病例任务详情 | P0 | 展示病例相关任务的关键信息，包括产品、国家地区、严重性、起算时间、随访状态、医学意见和附件。 | 用户能从任务详情进入评论、附件、复核和审计记录。 |
| F7 | 审计日志 | P0 | 记录关键操作：创建、编辑、状态变更、附件上传、评论、复核、导出和权限变更。 | 日志包含操作者、时间、对象、动作和变更前后信息。 |
| F8 | 审计包导出 | P0 | 按项目导出任务清单、附件清单、审计日志、审批记录和提交回执。MVP 先导出 HTML，PDF 延后。 | 导出文件能回答"谁在何时完成了什么，以及证据在哪里"。 |
| F9 | 供应商协作 | P1 | 为 CRO 或外部供应商提供受限项目空间，展示其负责的任务、SLA、缺失项和反馈意见。 | 供应商无法看到项目内其他敏感任务；可上传交付物并查看反馈。 |
| F10 | AI 工作助手 | P1 | 基于项目数据生成周报草稿、会议纪要草稿、风险摘要和 CAPA 初稿。所有 AI 内容默认标记为草稿，必须人工确认。 | AI 输出不得自动改变任务状态；用户确认后才可保存为正式记录。 |
| F11 | 信号主题池 | P2 | 汇总可能的安全性信号主题，关联病例、文献、会议结论和行动项。 | 作为后续版本，不影响 MVP 主流程上线。 |

## 关键业务规则

| 规则类型 | 规则说明 | 异常处理 |
|---|---|---|
| 权限 | 项目成员只能访问授权项目；供应商只能访问被分配任务；QA 可查看审计记录但不默认拥有系统配置权限。 | 无权限访问时提示"无访问权限"，不显示项目名称和敏感字段。 |
| 任务状态 | 任务必须按状态机流转。处理人可提交复核，复核人可通过或退回。 | 并发编辑时以后提交者收到冲突提示，需刷新后重试。 |
| 时限提醒 | 任务到期前、逾期后、复核退回时生成通知。提醒阈值由项目模板配置。 | 若截止时间为空，不允许发布任务；若负责人为空，任务进入"待分配"。 |
| 附件 | 附件必须关联项目和任务，记录上传人、上传时间和版本。删除附件需要保留删除记录。 | 敏感附件下载需校验权限；下载失败可重试。 |
| AI 输出 | AI 只生成草稿和摘要，不直接改变任务状态，不替代医学判断，不自动外发。 | 低置信度或缺少上下文时提示用户补充信息，而不是强行生成结论。 |
| 审计日志 | 关键操作均写入日志，不允许普通用户修改或删除。 | 日志写入失败时关键操作应失败或进入待重试状态，具体策略待技术评审确认。 |

## 数据模型

以下为产品层数据模型，定义了 7 个核心实体及其字段、类型和约束。不指定具体数据库选型，供前后端开发参考。

### 实体关系

```
User ||--o{ Project : "创建/管理"
User ||--o{ Task : "负责"
Project ||--o{ Task : "包含"
Project ||--o{ AuditLog : "产生"
Task ||--o{ Comment : "包含"
Task ||--o{ Attachment : "包含"
Template ||--o{ Project : "基于"
Task ||--o{ AuditLog : "记录"
```

### User（用户）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | string (UUID) | PK | 用户唯一标识 |
| name | string | 必填，最长 40 字 | 显示名称 |
| role | enum | 必填 | pm / processor / physician / qa / vendor |
| email | string | 可选 | 预留，MVP 不启用邮件 |
| createdAt | datetime | 必填 | 创建时间 |

### Project（项目）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | string (UUID) | PK | 项目唯一标识 |
| name | string | 必填，最长 60 字 | 项目名称 |
| product | string | 必填 | 产品线 |
| region | string | 必填 | 国家/地区 |
| type | enum | 必填 | ICSR / CAPA / 监管问询 / 信号评估 |
| status | enum | 必填，默认 进行中 | 未开始 / 进行中 / 已完成 / 已归档 |
| createdBy | string (UUID) | FK → User | 创建者 |
| createdAt | datetime | 必填 | 创建时间 |
| updatedAt | datetime | 必填 | 更新时间 |

### Task（任务）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | string (UUID) | PK | 任务唯一标识 |
| projectId | string (UUID) | FK → Project | 所属项目 |
| title | string | 必填，最长 120 字 | 任务标题 |
| description | string | 可选 | 任务描述 |
| assigneeId | string (UUID) | FK → User | 负责人 |
| status | enum | 必填，默认 未开始 | 未开始 / 处理中 / 待复核 / 需补充 / 已完成 |
| risk | enum | 必填，默认 中 | 高 / 中 / 低 |
| dueDate | date | 必填 | 截止时间 |
| evidenceRequired | string | 可选 | 必填证据描述 |
| caseNumber | string | 可选 | 关联病例编号 |
| createdAt | datetime | 必填 | 创建时间 |
| updatedAt | datetime | 必填 | 更新时间 |

### Comment（评论）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | string (UUID) | PK | 评论唯一标识 |
| taskId | string (UUID) | FK → Task | 所属任务 |
| authorId | string (UUID) | FK → User | 作者 |
| content | string | 必填，最长 2000 字 | 评论内容 |
| createdAt | datetime | 必填 | 创建时间 |

### Attachment（附件）

| 字段 | 类型 | 约束 | 说明 |

### AuditLog（审计日志）

| actorRole | enum | 必填 | 操作者当时的角色 |
| oldValue | JSON | 可选 | 变更前数据快照 |
| newValue | JSON | 可选 | 变更后数据快照 |

### Template（项目模板）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|

## API 接口设计

以下 API 采用 RESTful 风格，请求/响应格式为 JSON。认证方式 MVP 阶段可用简单 Token，后续升级为 JWT。

### 项目相关

| 方法 | 端点 | 说明 |
|---|---|---|
| GET | /api/projects | 项目列表（支持分页、筛选、搜索） |
| POST | /api/projects | 创建项目（基于模板自动生成任务） |
| GET | /api/projects/:id | 项目详情 |
| PUT | /api/projects/:id | 更新项目 |
| DELETE | /api/projects/:id | 归档项目（软删除） |

### 任务相关

| 方法 | 端点 | 说明 |
|---|---|---|
| GET | /api/tasks | 任务列表（支持按项目/状态/风险/负责人筛选） |
| GET | /api/tasks/:id | 任务详情（含评论、附件、审计记录） |
| PUT | /api/tasks/:id/status | 更新任务状态（触发审计日志） |
| PUT | /api/tasks/:id/assign | 分配/重分配任务 |
| POST | /api/tasks/:id/comments | 添加评论 |
| POST | /api/tasks/:id/attachments | 上传附件（触发审计日志） |

### 审计相关

| 方法 | 端点 | 说明 |
|---|---|---|
| GET | /api/audit-logs | 审计日志列表（支持按项目/任务/操作者/时间范围筛选） |
| GET | /api/projects/:id/audit-export | 导出项目审计包（HTML 格式） |

### 模板相关

| 方法 | 端点 | 说明 |
|---|---|---|
| GET | /api/templates | 模板列表 |
| GET | /api/templates/:id | 模板详情 |

### 通用响应格式

**成功响应**：
```json
{
  "success": true,
  "data": {},
  "message": "操作成功"
}
```

**错误响应**：
```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "任务不存在"
  }
}
```

## 指标与埋点

MVP 阶段的指标用于判断产品是否真正减少 PV 项目管理中的重复劳动和遗漏风险。

### 北极星指标

| 指标名称 | 定义 | 计算方式 | 目标建议 |
|---|---|---|---|
| 任务按时完成率 | 衡量团队是否能在监管时限内完成任务 | 已完成且完成时间 ≤ 截止时间的任务数 / 已完成任务总数 × 100% | MVP 阶段 ≥ 70%，稳定后 ≥ 85% |

### 过程指标

| 指标名称 | 定义 | 计算方式 | 用途 |
|---|---|---|---|
| 任务复核退回率 | 衡量初稿质量 | 进入"需补充"状态的任务数 / 提交复核的任务总数 × 100% | 判断模板设计和培训是否到位 |
| 审计证据完整度 | 衡量审计包可用性 | 必填证据已上传的任务数 / 有必填证据要求的任务总数 × 100% | 判断审计流程是否被执行 |
| 高风险任务逾期率 | 衡量紧急事项响应 | 逾期的"高风险"任务数 / 高风险任务总数 × 100% | 判断驾驶舱和提醒是否有效 |

### 暂不纳入 MVP 的指标

- 周报生成使用率（需 AI 模块上线后统计）
- 供应商延迟任务数（需供应商协作模块完善后统计）
- 用户活跃留存率（需真实用户数据积累）

## 非功能需求

| 维度 | 需求 | 说明 |
|---|---|---|
| 安全 | 权限与数据隔离 | 必须支持租户级隔离、角色级访问控制和敏感附件权限校验。供应商账号不得越权查看其他项目。 |
| 审计 | 关键操作留痕 | 任务、附件、复核、导出、权限变更等操作必须写入审计日志，并可按项目导出。 |
| 性能 | 常用页面响应 | 项目驾驶舱、任务列表和任务详情应在常规数据量下保持可用响应。具体性能指标待技术评审补充。 |
| 可用性 | 异常状态清晰 | 空数据、无权限、附件上传失败、AI 生成失败、网络中断等情况必须有明确提示和可恢复路径。 |

## 验收标准

| 验收项 | 通过标准 |
|---|---|
| 主流程可完成 | 用户可以从模板创建项目，自动生成任务，分配负责人，处理任务，提交复核，完成任务，并导出审计包。 |
| 权限正确 | 项目成员、QA、供应商、管理员看到的数据和可执行操作符合角色权限。 |
| 状态流转正确 | 任务状态只能按规则流转；复核退回必须填写原因；关键状态变更写入审计日志。 |
| 时限提醒可用 | 临近截止、逾期、复核退回、证据缺失能够产生站内通知，并在驾驶舱展示。 |
| 审计包可导出 | 按项目导出的文件包含任务清单、附件清单、复核记录和审计日志。 |
| AI 不越界 | AI 输出均为草稿，不自动改变任务状态，不自动外发，不生成医学定论。 |

## 决策记录与后续迭代

### 已确认决策

| 事项 | 决策 | 理由 |
|---|---|---|
| 首批 MVP 用户 | 药企内部 PV 团队 | 当前角色模型和权限体系已覆盖此场景 |
| 是否接入现有安全数据库 | MVP 不接入，采用独立系统 | 降低集成复杂度，demo 自包含；后续通过 API 扩展 |
| 法规时限规则 | 先做可配置相对时限 | 不在 MVP 中内置完整全球规则库，避免合规风险 |
| AI 输出边界 | AI 只做草稿、摘要、会议纪要和风险提示 | 必须人工确认，不替代医学判断 |
| 审计包格式 | MVP 优先 HTML，PDF 延后 | HTML 导出已实现，PDF 需要额外库 |

### 后续迭代建议

1. **接入后端**：按本文 API 定义实现 Node.js/Python 后端，替换 localStorage
2. **用户认证**：接入企业 SSO 或实现基于邮箱的登录
3. **邮件提醒**：补充 SMTP 服务，实现逾期/复核提醒邮件
4. **PDF 导出**：引入 pdf-lib 或 puppeteer，支持审计包 PDF 导出
5. **AI 助手**：接入 LLM API，实现周报/会议纪要草稿生成
6. **移动端适配**：补充响应式布局或开发微信小程序

## 版本规划

| 阶段 | 目标 | 范围 | 进入下一阶段条件 |
|---|---|---|---|
| MVP | 跑通 PV 项目管理主链路。 | 项目、模板、任务、时限、附件、评论、复核、审计日志、审计包。 | 试点团队能用它管理真实或模拟项目，并完成审计导出。 |
| Beta | 提升团队协作和汇报效率。 | 供应商协作、CAPA、AI 周报、会议纪要、风险摘要。 | AI 草稿被用户接受并编辑使用，供应商协作权限验证通过。 |
| 正式版 | 覆盖更完整的 PV 项目类型。 | 信号主题池、聚合报告、数据看板、API 集成、多租户配置。 | 核心客户确认项目类型和集成优先级。 |
