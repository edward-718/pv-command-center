# PV智枢 第二阶段 PRD

> 版本：v1.0
> 日期：2026-06-29
> 范围：MedDRA 医学评估 + 随访管理 + 监管提交追踪 + 任务依赖关系
> 前置条件：第一阶段已完成（数据结构扩展、法规时限引擎、通知中心、标签页重构）

---

## 一、需求背景

第一阶段已完成：
- 数据层扩展（Project/Task 新增字段、notifications/activities 顶层数据）
- 法规时限引擎（Day 0 + 法规规则自动计算截止日）
- 通知中心（逾期/临近/复核退回等提醒）
- 任务详情标签页重构（五标签页框架：评论/随访/医学评估/监管提交/证据与日志）

当前「随访」「医学评估」「监管提交」三个标签页为占位状态，仅显示「功能将在第二阶段上线」提示。第二阶段目标：将这三个标签页从占位变为真实功能，同时实现任务依赖的前置阻塞逻辑，完成从「病例获知」到「监管提交」的完整 ICSR 闭环。

## 三、用户故事

### 故事 1：Safety Physician 填写医学评估

> 作为 Safety Physician，我需要在任务详情中填写严重性、因果关系和 MedDRA 编码，并给出医学审评结论。

**验收标准**：
- 医学评估标签页显示可编辑表单（非占位提示）
- 表单字段：严重性（单选）、因果关系（单选）、MedDRA PT 码（文本）、MedDRA LLT 码（文本）、医学审评结论（多行文本）、需升级信号（复选框）
- 只有 Physician 和 PM 角色可编辑；其他角色只读
- 保存后数据写入 Task 对象，刷新不丢失
- 医学评估变更记录到审计日志

### 故事 2：Processor 创建随访任务

> 作为 Case Processor，我需要在收到随访信息后，基于现有任务创建随访任务，记录随访变更内容。

**验收标准**：
- 随访标签页显示该病例的所有随访轮次（按时间倒序）
- 每轮随访显示：轮次号、随访日期、随访来源、变更内容摘要
- 点击「创建随访」按钮，基于当前任务生成下一轮随访任务
- 新任务标题自动带「[随访-N]」前缀（如「[随访-1] 病例建档与起算点确认」）
- 随访任务继承原任务的产品、地区、病例编号等基础信息
- 随访任务可独立流转状态（未开始→处理中→待复核→已完成）
- 项目卡片显示随访轮次统计（如「随访 1/2」）

### 故事 3：项目经理记录监管提交

> 作为 PV 项目经理，我需要记录向监管机构提交的信息，包括提交渠道、回执编号和回执日期。

**验收标准**：
- 监管提交标签页显示提交记录列表（按时间倒序）
- 支持添加提交记录：监管机构、提交渠道、提交日期、回执编号、回执日期、状态、备注
- 提交记录保存后，项目卡片显示「已提交」标记
- 提交状态变更（待提交→已提交→已确认/被退回）记录到审计日志
- 支持编辑和删除提交记录（删除需确认）

### 故事 4：Processor 处理有依赖关系的任务

> 作为 Case Processor，我需要清楚知道哪些任务因为前置任务未完成而被阻塞，避免提前处理。

**验收标准**：
- 任务列表中被阻塞的任务显示「阻塞」标签，操作按钮禁用
- 任务详情中显示「前置任务」列表（可点击跳转）和「后续任务」列表
- 前置任务状态变为「已完成」时，后续任务自动解除阻塞
- 阻塞解除时，系统生成通知告知后续任务负责人
- 创建项目时，如果模板配置了任务依赖，自动生成依赖关系

## 五、业务规则

### 5.1 MedDRA 字段编辑权限

| 字段 | 可编辑角色 | 只读角色 |
|---|---|---|
| 严重性 | Physician, PM | Processor, QA, Vendor |
| 因果关系 | Physician, PM | Processor, QA, Vendor |
| MedDRA PT | Physician, PM | Processor, QA, Vendor |
| MedDRA LLT | Physician, PM | Processor, QA, Vendor |
| 医学审评结论 | Physician, PM | Processor, QA, Vendor |
| 需升级信号 | Physician, PM | Processor, QA, Vendor |

**规则**：非 Physician/PM 角色打开医学评估标签时，表单为只读状态，不显示保存按钮。

### 5.2 随访管理规则

| 规则 | 说明 |
|---|---|
| 创建权限 | PM 和 Processor 可创建随访任务 |
| 轮次限制 | 不限制随访轮次，按实际业务需求创建 |
| 状态独立 | 随访任务有独立的状态流转，不依赖原任务状态 |
| 依赖关系 | 随访任务默认依赖原任务（原任务需完成，随访任务才能开始） |
| 字段继承 | 产品、地区、病例编号继承原任务；医学字段不继承，需重新评估 |
| 通知触发 | 随访任务创建时，通知分配给负责人；随访任务逾期时，按普通任务规则生成通知 |

### 5.3 监管提交追踪规则

| 规则 | 说明 |
|---|---|
| 添加权限 | PM 和 Processor 可添加提交记录 |
| 编辑权限 | 只有创建者可编辑自己添加的提交记录 |
| 删除权限 | PM 可删除任何提交记录；创建者可删除自己的记录 |
| 状态流转 | 待提交 → 已提交 → 已确认 / 被退回 |
| 通知触发 | 提交记录状态变为「被退回」时，通知 PM 和任务负责人 |

### 5.4 任务依赖规则

| 规则 | 说明 |
|---|---|
| 依赖配置 | 在模板节点中配置 `dependsOn` 字段，指定前置节点 index |
| 自动生成 | 创建项目时，根据模板依赖配置自动生成任务间的 `dependsOn` 关系 |
| 阻塞条件 | 任务 A 依赖任务 B，任务 B 状态不是「已完成」时，任务 A 被阻塞 |
| 自动解除 | 任务 B 状态变为「已完成」时，自动检查并解除任务 A 的阻塞 |
| 阻塞展示 | 被阻塞的任务显示「阻塞」标签，操作按钮禁用，详情页显示前置任务链接 |
| 通知触发 | 阻塞解除时，通知后续任务负责人 |

| 阻塞状态持久化 | 任务阻塞状态保存在 localStorage 中，刷新后正确恢复 |

## 十、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 医学评估字段过多导致表单过长 | 中 | 使用折叠面板分组（基础信息 / MedDRA 编码 / 审评结论），默认展开基础信息 |
| 随访任务无限创建导致数据膨胀 | 中 | 不限制创建次数（符合真实业务），但 UI 上只展示最近 10 轮，更早的折叠 |
| 提交记录删除后审计追踪断裂 | 中 | 删除操作记录审计日志（包含被删除记录的完整数据快照），实现「逻辑删除」而非物理删除 |
| 任务依赖循环（A 依赖 B，B 依赖 A） | 低 | 创建项目时检测循环依赖，发现时抛出错误并阻止项目创建 |
| 阻塞解除通知刷屏 | 低 | 如果一个前置任务完成同时解除多个后续任务，合并为一条通知 |
